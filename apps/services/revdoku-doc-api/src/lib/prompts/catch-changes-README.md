# Change Detection Description Format Spec

This document defines the standardized format for change detection descriptions
generated by the catch-changes change detection rule.

## Format Templates

| Action  | With location                          | Without location              |
|---------|----------------------------------------|-------------------------------|
| Changed | `<Where> changed "<prev>" to "<curr>"` | `Changed "<prev>" to "<curr>"`|
| Added   | `<Where> added "<content>"`            | `Added "<content>"`           |
| Removed | `<Where> removed "<content>"`          | `Removed "<content>"`         |

## Rules

- Always use double quotes around values (`"..."`)
- No trailing period
- `<Where>` is a short location label (e.g. "Invoice number", "Professional Fees: Rate for third line item")
- One change per description -- never combine multiple changes

## Examples

```
Header text changed "INVOICE" to "REVISED INVOICE"
Invoice number changed "GL-2026-0024" to "GL-2026-0024-R1"
Changed "$1,575.00" to "$2,250.00"
Professional Fees: Title for third line item changed "Associate" to "Senior Partner"
Added "Replaces: GL-2026-0024"
Footer added "Page 1 of 2"
Payment terms removed "Net 30"
Removed "Confidential Draft"
```

## Relationship to Structured Fields

The AI also populates `val_p` and `val` structured fields directly:
- `val_p`: exact previous text/value (empty string if new content)
- `val`: exact current text/value (empty string if removed)

These structured fields are the primary source of truth. The description format
serves as a human-readable summary AND as a fallback extraction source when
`val_p`/`val` are missing.

## Fallback Extraction

`check-data-utils.ts` contains `extractPrevCurrentFromDescription()` which
parses descriptions to extract `val_p`/`val` when the AI omits them. The
function matches the standard formats first, then falls back to legacy patterns
for backward compatibility with older descriptions.
