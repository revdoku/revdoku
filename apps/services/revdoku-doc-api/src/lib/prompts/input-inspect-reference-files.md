## REFERENCE FILES
- **Reference files ARE active inspection inputs. You MUST cross-reference them.** One or more `[REFERENCE #file_N ...]` blocks are present in this prompt — these are authoritative source-of-truth documents (engagement letters, quotes, purchase orders, rate cards, contracts, prior invoices) that the user attached for this specific review.
- **FIRST — verify the reference applies to THIS document.** Before you rely on any attached reference, confirm it is for the same subject as the document under review. Cross-check vendor/provider name, account/customer identifier, billing period, jurisdiction, matter/project number, or any other key identifier that would tell you these two documents belong together. If the reference is clearly for a DIFFERENT vendor, a DIFFERENT account, a DIFFERENT billing period, or a DIFFERENT scope (e.g. an AWS billing policy for account 12345 was attached but the invoice is for account 67890; a quote for Project Alpha was attached but the invoice is for Project Beta; a prior-year rate card was attached but the invoice is from this year), then the reference does NOT apply. In that case:
  - Emit a dedicated FAILED check against whichever checklist rule most closely relates to "reference applicability" (or the first rule if none does), with `val` set to the mismatched identifier (e.g. `val="account 12345 vs invoice account 67890"`) and the description clearly stating the mismatch and naming the attached reference file — e.g. "Attached policy `account-department-policy.pdf` is for account 12345 but invoice is for account 67890 — policy does not apply to this invoice (#file_1)."
  - Do NOT pretend the reference applies. Do NOT force its rate card, category list, timekeepers, or rules onto the primary document when the identifiers don't match.
  - You can still cite `(#file_N)` in that failure message so the reader can jump to the mismatch in the ref viewer.
  - If the reference DOES apply, no applicability check is needed — just proceed with normal cross-referencing below.
- For **every rule** in the checklist, ask yourself: "Does any attached reference file say something about this?" If yes, perform the comparison and emit an additional check in that rule's `checks` array for each deviation you find — even if the rule's plain text does not explicitly mention the reference. A rule with reference-relevant content produces MULTIPLE checks (one per deviation). A single "passed" check per rule when references were available is a failure mode.
- Typical deviations to flag when you detect them, regardless of whether a rule names them: unapproved people/vendors/items, rates above the authorized rate, prohibited categories being billed (admin tasks, training, etc.), caps being exceeded (%/$/count), terms violated (billing window, travel %, receipts, approvals), any line item not supported by the reference.
- Silent passes on rules that should have been cross-referenced against an attached file will be penalized. If the attached file is irrelevant to a rule, that's fine — just skip. If it's relevant, compare and cite.
- Each reference file is split into per-page blocks of the form `[REFERENCE #file_N page=P] ... [END OF REFERENCE PAGE]`, where `P` is a 0-indexed page number within that reference file.
- REQUIRED: Every check whose determination relies on data from a reference file MUST include the file citation token in parentheses — e.g. "(#file_1)". Cite on BOTH passed and failed checks whenever comparison against a reference file is what produced the conclusion. Place the token at the end of the sentence it supports.
- **HARD RULE — no empty `ref`:** whenever the check description cites a `#file_N` token, the check's `data.ref` field MUST be a non-empty string AND `data.ref_page` MUST be set. If you cannot produce a verbatim locator from the reference file text, either (a) the check doesn't actually depend on the ref — remove the `(#file_N)` citation, or (b) find SOMETHING verbatim to anchor on (section header, first words of a paragraph, column label). An empty `ref` next to a `#file_N` citation is never acceptable.
- Cite multiple files when a check spans more than one reference: "...which does not match the quoted quantity of 280 sq ft (#file_1) or the PO quantity of 290 sq ft (#file_2)."
- The `ref` field MUST be a **verbatim substring copied from the reference file text** — exactly as it appears there, including units, currency symbols, percent signs, quotes, punctuation. It is the locator for a human to find the authoritative source. DO NOT put a derived, computed, or paraphrased value in `ref`. Put the computed value in the description prose instead.
  - Correct: rule says "fees must not exceed 8% of total"; invoice shows $2,032.50 (15.8%) over $12,887.50 total; engagement letter literally says "shall not exceed **8% of total invoice fees**". → `val="$2,032.50"`, `ref="8%"`, description = "Internal firm conference fees total $2,032.50 (15.8% of fees), exceeding the 8% cap ($1,031.00) agreed in the engagement letter (#file_1)."
  - WRONG: `ref="$1,031.00"` — that is the *computed* cap, not text in the reference file.
  - Correct: rule says "travel time is billed at 50%"; engagement letter says "Travel time shall be billed at **50% of the timekeeper's standard hourly rate**"; invoice shows $485 travel. → `val="$485.00"`, `ref="50% of the timekeeper's standard hourly rate"` (or shorter verbatim fragment "50%"), description = "Travel billed at full $485.00 instead of the agreed 50% of the timekeeper's standard hourly rate ($242.50) (#file_1)."
  - WRONG: `ref="$242.50"` — that is the computed expected amount, not in the reference.
- Whenever you set `ref`, you MUST also set `ref_page` to the `page=P` number of the block where the verbatim substring appears. Do NOT guess — echo the `P` from the `[REFERENCE #file_N page=P]` marker verbatim. If the value spans multiple pages, use the first page.
- When the reference file contains an explicit numeric value that IS the right-hand side of the comparison (e.g. the quote lists "Quantity: **280 sq ft**" and the rule checks quantity matches), then `ref` is that verbatim number with its unit: `ref="280 sq ft"`, `ref_page=1`. That's fine — the key test is "did I copy this string from the ref text?" vs "did I calculate it?".
- **`ref` must NOT be empty whenever you cite a `#file_N` token.** If the rule compared against a *list, set, or section* rather than a single value (e.g. "Is this timekeeper in the approved-timekeepers list?", "Does this action require written approval under the engagement terms?"), set `ref` to a verbatim locator that points a human to the relevant section in the reference file. Best locator, in priority order:
   1. The exact item/row being referenced if it's in the list — e.g. `ref="James Okafor — $485.00 / hr"` (verbatim row from the table).
   2. The section header of the block the rule consulted — e.g. `ref="Approved Timekeepers and Rate Card"` when timekeeper authorization was the check.
   3. The first ~40 characters of the most relevant paragraph — e.g. `ref="Only the timekeepers listed below are authorized to bill"`.
   Still set `ref_page` to the page where the locator appears.
   - Correct (list/set rule): rule asks "is T. Nguyen in the approved timekeepers list?"; engagement letter lists Patricia Whitfield, James Okafor, Rachel Kim, Paralegal Staff under a "Approved Timekeepers and Rate Card" header on page 0. T. Nguyen is not listed. → `val="T. Nguyen"`, `ref="Approved Timekeepers and Rate Card"` (or the list rows as a single string), `ref_page=0`, description = "Line item for T. Nguyen on 03/12 is for an unapproved timekeeper — engagement letter's Approved Timekeepers and Rate Card lists only Patricia Whitfield, James Okafor, Rachel Kim, and Paralegal Staff (#file_1)."
   - WRONG: `ref=""` — always give the reader a pointer back to the ref text, even for set-membership checks.
- Do NOT cite a file token when the check did not actually consult that reference. No fabricated citations.
